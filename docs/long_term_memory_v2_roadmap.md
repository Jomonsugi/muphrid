## What v1 Does and Doesn't Do

### v1 delivers:
- HITL-only memory extraction (observations, failures, preferences)
- SQLite + sqlite-vec + FTS5 hybrid search with RRF fusion
- Qwen3-Embedding-8B via Ollama with SHA256 cache
- `memory_search` tool available to the agent in all phases
- Feature flag (MEMORY_ENABLED) — zero overhead when off
- `source` column for tagging memory provenance
- `confidence`, `reinforcement_count`, `valid_until` columns in schema (unused)
- Automatic embedding backfill on startup

### v1 does NOT do:
- Autonomous/phase gate memory extraction
- Confidence updates (reinforcement or contradiction detection)
- Memory consolidation or deduplication
- Memory expiration or TTL
- Session-level records (the `sessions` table exists but isn't populated in v1)
- Graph memory or entity relationships
- Memory inspection UI
- Prompt optimization from memory patterns

## Additional Memory Sources

### The `source` column is free-form text — adding new sources is a code change, not a schema migration.

### Phase gate extraction (`source="phase_gate"`)
**Research basis:** Oracle MemoryManager, OpenClaw (Lesson #5)

The `processing_log.md` is already generated by `_write_phase_report()` in
`t30_advance_phase.py` at every phase transition. It contains the agent's full
reasoning, every tool call with parameters, and outcomes.

**What to build:**
- After `_write_phase_report()` in `advance_phase`, read the phase section just written
- Call `extract_phase_learnings(phase_log_section, session_context, llm)` (function
  signature already defined in `extraction.py`)
- Store results with `source="phase_gate"`
- Search results prioritize `hitl` over `phase_gate`

**When to add:** When the agent's autonomous decisions are good enough that you'd want
to learn from them. The risk is the "experience-following" problem (Lesson #9) — the
agent repeating its own mistakes because they're in memory.

**Schema ready:** Yes — `source` column accepts any string.

### analyze_image reasoning (`source="analyze"`)
**Potential value:** The `analyze_image` tool returns rich metrics and the agent's
reasoning about those metrics could be valuable. "When SNR dropped below 50 after
denoising, the strength was too high" is a learning that comes from the agent's
interpretation of analyze_image output.

**What to build:** Hook into the agent node after analyze_image tool calls. Extract
the agent's reasoning about the metrics (from the AIMessage following the ToolMessage).

**When to add:** After phase gate extraction is proven. This is higher volume and
lower signal — needs careful filtering to avoid noise.

### Tool-specific extraction (`source="tool_call"`)
**Potential value:** Some tools produce particularly informative outputs (e.g.,
`color_calibrate` reports plate-solve-derived focal length, `stretch_image` reports
clipping percentages). Capturing the raw tool output + agent's response could be
valuable for specific tools.

**When to add:** Only if analysis shows specific tool interactions are consistently
useful across sessions.

## Confidence & Reinforcement Logic

### Research basis: Hindsight (Lesson #3)

The schema has `confidence REAL DEFAULT 0.5` and `reinforcement_count INTEGER DEFAULT 1`
on `observations` and `preferences` tables. v1 stores defaults; v2 adds update logic.

### Reinforcement detection
When a new memory is extracted, check for semantically similar existing memories
(cosine similarity > threshold, same phase, compatible context):
```
existing.confidence = min(existing.confidence + 0.1, 1.0)
existing.reinforcement_count += 1
existing.updated_at = now
# Skip storing the new memory (it's redundant)
```

### Contradiction detection
When a new memory contradicts an existing one (similar context but conflicting content):
```
existing.confidence = max(existing.confidence - 0.2, 0.0)
if existing.confidence == 0.0:
    existing.valid_until = now  # invalidate
# Store the new memory at confidence 0.5
```

### Asymmetric updates (from Hindsight)
Contradictions hit harder (-0.2) than reinforcements (+0.1). This prevents the agent
from becoming overconfident about patterns that may be coincidental.

### How to detect similarity/contradiction
- **Similarity:** Embed both memories, compute cosine similarity. Threshold ~0.85 for
  "same observation" vs ~0.7 for "related."
- **Contradiction:** Harder. Options:
  1. LLM-based: Ask the extraction LLM "does this new observation contradict any of
     these existing memories?" (adds LLM calls)
  2. Heuristic: Same tool + same phase + significantly different parameters = potential
     contradiction. Flag for review rather than auto-invalidating.

### When to add
When you have 50+ memories and start noticing redundancy or contradictions in search
results. The backfill script can retroactively compute similarity between existing
memories.

## Memory Consolidation & Bloat Prevention

### Research basis: Mem0, Zep, Academic (memory bloat patterns)

### The problem
After 50 sessions on emission nebulae, the store will have dozens of observations
saying variations of "GHS D=10 works well for compressed Bayer data." Search results
become noisy with near-duplicates.

### Consolidation strategy
Periodically merge semantically similar memories into single stronger entries:
```
Memory 1: "GHS D=10, B=6, SP=0.007 worked well on M20 emission nebula"
Memory 2: "GHS D=10, B=5, SP=0.008 produced good stretch on NGC7000 nebula"
Memory 3: "GHS D=10, B=7, SP=0.006 was effective on IC1396 emission nebula"

CONSOLIDATE INTO:
"GHS D=10 with B=5-7 and SP=0.006-0.008 consistently produces good stretches
 on emission nebulae (validated across M20, NGC7000, IC1396). Higher B focuses
 the stretch more tightly around the symmetry point."
 confidence: 0.8, reinforcement_count: 3
```

### TTL / expiration
- Memories unreinforced for N sessions could have confidence decayed
- Old memories with low confidence could be auto-invalidated
- Episodic details (specific session parameters) are less valuable over time
  than generalized patterns

### Deduplication at write time
Before storing a new memory, check if a near-duplicate already exists
(cosine similarity > 0.9). If so, reinforce instead of adding.

### Critical warning from research
Unlike API errors (explicit), memory eviction that removes the wrong records
**silently degrades quality** — no exception, just worse responses. Consolidation
must be conservative and logged.

### When to add
When `MemoryStore.stats()` shows 100+ observations and you notice redundancy in
search results. Start with deduplication at write time (simplest), then add
periodic consolidation.

## Phase Gate Extraction

### Full design (deferred from v1)

**Trigger:** Inside `advance_phase` (T30), after `_write_phase_report()`:
```python
if is_memory_enabled() and is_phase_gate_extraction_enabled():
    phase_log_section = _read_latest_phase_section(working_dir)
    extraction = extract_phase_learnings(phase_log_section, session_context, llm)
    for obs in extraction.observations:
        store.add_observation(..., source="phase_gate")
    for fail in extraction.failures:
        store.add_failure(..., source="phase_gate")
```

**The `extract_phase_learnings` function:**
- Already defined in extraction.py as a v2 signature
- Takes the processing_log.md phase section as input
- LLM extracts learnings from the full tool call sequence
- More context than HITL extraction (sees all tool calls, not just the HITL conversation)
- Risk: may extract learnings from bad autonomous decisions (Lesson #9)

**Search prioritization:**
When both sources exist, search should rank `hitl` above `phase_gate`. Simple
approach: add a small RRF bonus for hitl source (already in store.py search method).

**Flag consideration:**
Could be a second env var (`MEMORY_PHASE_GATE_ENABLED=false`) or just a code change.
Separate flag gives the user control without modifying code.

## Session Records

### The `sessions` table exists but isn't populated in v1.

**What to store:**
- Thread ID, target name, target type, sensor, conditions (bortle, SQM)
- Integration time, frame count
- Outcome (success/partial/failed), final SNR
- Processing time

**When to write:**
- At end of run (EXPORT → COMPLETE transition in advance_phase)
- Or at start of run (partial record) and updated at completion

**Value:**
- Enables queries like "show me all sessions on galaxies with ASI2600MC"
- Session context enriches individual memory search results (already supported —
  `_load_memory` joins to sessions table)
- Track success rate over time

**When to add:** Alongside phase gate extraction — both fire from advance_phase.

## Search Improvements

### Recency boosting
Current search ranks by RRF score only. Adding a recency boost would prefer
recent memories over old ones when relevance is similar:
```python
recency_weight = 1.0 / (1 + days_since_created * decay_rate)
final_score = rrf_score + recency_boost * recency_weight
```

### Confidence-weighted ranking
When confidence logic is active, search should boost high-confidence memories:
```python
final_score = rrf_score * (0.5 + 0.5 * confidence)
```

### Cross-encoder reranking (from Hindsight)
After RRF fusion, take top-50 results and rerank with a cross-encoder model
for more precise relevance. Adds latency but significantly improves precision.
Likely overkill until the memory store is large.

### Filter by session metadata
The `memory_search` tool currently accepts `memory_type`, `target_type`, and `phase`
filters. Future filters:
- `sensor` — "what worked with this camera?"
- `bortle` range — "what worked in similar light pollution?"
- `date_range` — "what have I learned recently?"
- `min_confidence` — "only show me validated patterns"

### Search result formatting
As the store grows, the output format may need:
- Truncation of long parameter dicts
- Grouping by type (all observations, then failures, then preferences)
- Summary statistics ("12 memories found, showing top 5")

## Embedding Model Considerations

### Model choice
v1 uses Qwen3-Embedding-8B via Ollama. Considerations for the future:

- **Dimension:** 4096 floats per embedding. If storage becomes an issue, consider
  a smaller model (nomic-embed-text at 768 dims, BGE-small at 384 dims).
- **Speed:** 8B model is fast on Apple Silicon but slower than smaller models.
  For backfilling hundreds of memories, a smaller model may be preferable.
- **Model upgrades:** If switching models, all existing embeddings need re-computing.
  The embedding_cache keys include the model name, so old cache entries won't collide.
  But memory_vec entries would need full re-embedding.

### Offline operation
If Ollama goes down mid-session, the embedder returns None and search falls back to
FTS5-only. This is graceful but means search quality temporarily degrades. Consider
caching recent query embeddings in memory (not just in the SQLite cache) for faster
recovery.

### Alternative embedding approaches
- **sentence-transformers:** Run embedding model directly in-process (no Ollama server).
  Adds PyTorch as a dependency but removes the server requirement.
- **API-based:** Use OpenAI/Voyage/Cohere embeddings for higher quality. Adds network
  dependency and cost but no local compute.

## Prompt Optimization as Procedural Memory

### Research basis: langmem (Lesson #6)

langmem treats the **system prompt itself** as a memory type. The prompt optimizer
analyzes conversation trajectories (especially failures) and generates refined
prompt instructions.

### How it could work for AstroAgent
After N sessions, analyze patterns in failures table:
- "The agent consistently uses too-conservative stretch parameters on first attempt"
- "The agent forgets to check clipping after stretch"
- Generate prompt additions: "When stretching, start with D=8 or higher for compressed
  Bayer data. Always check clipped_highlights_pct after each stretch variant."

### Implementation approach
1. Periodic analysis of failure patterns (manual or automated)
2. Generate prompt refinements via LLM
3. Store as a special memory type or append to SYSTEM_BASE
4. Version the prompt changes for rollback

### When to add
After 20+ sessions with meaningful failure patterns. This is sophisticated and
should be validated manually before automating.

## Graph Memory

### Research basis: Mem0^g, Cognee, Zep/Graphiti

### The case for graph memory in astrophotography
Astrophotography has rich entity relationships:
```
(M42) --[is_a]--> (emission_nebula)
(emission_nebula) --[benefits_from]--> (hue_targeted_saturation)
(ASI2600MC) --[sensor_type]--> (bayer)
(bayer) --[requires]--> (debayer=True)
(M42_session_001) --[used_stretch]--> (GHS D=10, B=6)
(GHS D=10) --[works_well_for]--> (compressed_bayer_data)
```

This would enable multi-hop reasoning: "What stretch parameters work for emission
nebulae shot with Bayer sensors?" → traverse graph rather than keyword search.

### Implementation options
1. **SQLite with adjacency tables:** Simple, no new dependencies. Two tables:
   `entities(id, name, type)` and `relationships(from_id, to_id, relation, confidence)`.
2. **Neo4j/Kuzu:** Purpose-built graph DB. More powerful queries but adds infrastructure.
3. **In-memory graph:** Build from SQLite at startup. Fast traversal, simple persistence.

### When to add
When the memory store has 200+ entries and you notice that keyword/vector search
misses relationship-based queries. Start with SQLite adjacency tables (option 1).

## Memory Inspection & Debugging Tools

### The store needs visibility

As the memory system grows, you'll want to inspect what's stored:

### CLI commands
```
astro-agent memory stats        # count by type, source, phase
astro-agent memory search "..."  # test search from command line
astro-agent memory list          # recent memories with metadata
astro-agent memory inspect 42    # full detail of a specific memory
astro-agent memory invalidate 42 # manually mark a memory invalid
astro-agent memory export        # dump to JSON for analysis
```

### Gradio tab
A "Memory" tab in the Gradio UI showing:
- Memory count by type
- Recent memories with search
- Ability to invalidate bad memories
- Search testing interface

### Logging
The current implementation logs memory operations at INFO level. Consider:
- A dedicated memory operation log file
- Structured logging (JSON) for memory events
- Tracking which memories were retrieved and used (did the agent act on them?)

### When to add
CLI inspection tools should come early (useful for debugging). Gradio tab can
wait until the memory system is proven.

## Backfill & Migration

### Embedding backfill (implemented in v1)
The `_backfill_embeddings()` method runs at startup and embeds any memories
missing from `memory_vec`. This handles the case where memories were saved
without Ollama running.

### Model migration
If switching embedding models (e.g., Qwen3 → nomic-embed-text):
1. Clear all entries from `memory_vec`
2. Clear `embedding_cache`
3. Restart — `_backfill_embeddings()` re-embeds everything with the new model
4. Update `memory_embedding_model` in .env

### Schema migration
If adding columns to existing tables in future versions:
- SQLite supports `ALTER TABLE ADD COLUMN` without rewriting data
- For virtual table changes (memory_vec, memory_fts), may need to drop and recreate
- Consider a `schema_version` table for tracking

### Data portability
The memory store is a single SQLite file (`~/.astro_agent/memory.db`). It can be:
- Copied between machines
- Backed up trivially
- Shared with other users (their agent benefits from your HITL teaching)
- Versioned alongside session data
