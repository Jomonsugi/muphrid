# Muphrid Audit — Can the Tools Support Pro-Level Output?

_April 18, 2026 · grounded in the M20 run `run-m20-20260417-165609` and full code inspection_

---

## 1. Bottom Line

**Yes — the open-source tool surface is capable of producing near-PixInsight output for typical deep-sky targets. It is not the primary bottleneck today.** The tools cover roughly 85% of a professional workflow; the remaining 15% can be bridged with a handful of targeted additions (polynomial gradient fallback, masked-apply wrapper, per-channel stretch, richer curves, HSV/HSL saturation). A skilled human operating _only_ with muphrid's tools could reach pro-adjacent results on M20 given enough integration time.

**The primary driver of the current quality gap is not the tools — it is the agent's ability to actually use them.** Specifically:

1. The prompt frames "quality" around _metrics holding steady_, not around _matching a visual reference_. The agent stops when numbers are okay, even when the picture is flat.
2. Key pro workflows (HDR masked stretch, selective multiscale sharpening with T27+T25, aggressive multi-pass saturation) are mentioned once or twice in prompts but never rehearsed as _this-is-what-pro-means_. They are treated as optional.
3. Feedback loops silently accept regressions (gradient_magnitude went from 0.012 → 0.428 and the agent moved on).
4. The checkpoint system has a semantic confusion (`save_checkpoint` saves state, not images; the agent misinterprets it as image-level branching) that caused LCE to be applied to the wrong file in the M20 run.
5. `deconvolution` was never called despite SNR=74.9 (the phase prompt explicitly says "only beneficial when snr_estimate > ~50").

Quality = tools × agent skill × workflow doctrine. The tools are strong. The agent is decent. **The workflow doctrine — encoded in prompts, tool docstrings, and the reward signal the agent measures against — is the weakest link.** This is also the cheapest layer to upgrade.

## 2. What Actually Happened on the M20 Run

The intermediate previews tell the story better than any metric. Putting them in order:

| Stage | Preview | Visual state |
|---|---|---|

The linear image, auto-stretched for display, contains **demonstrably more information** than the final product. The entire pipeline from T14 onward is a net-negative in terms of what a viewer sees. That is not a tools problem; it is a doctrine problem.

### Concrete deficits and root causes

| Deficit you named | What I also see | Root cause class | Specific cause |
|---|---|---|---|
| Blue reflection nebula barely visible | Yes — almost invisible in final; clearly present in linear autostretch preview | **Stretch + saturation doctrine** | V6 GHS with `HP=0.96` clipped the bright end too cautiously; only two hue-saturation passes, neither boosted blue reflection (hue_target=3 was applied but `amount=0.5` is mild); no luminance mask + T27 sharpening on the reflection region |
| Faint outer nebulosity largely missing | Yes — linear preview shows strong outer Hα field bridging M8↔M20; final has almost none | **Stretch doctrine** | GHS `SP=0.0125` is targeting the near-noise signal but the overall transform compressed it back. | Internal nebula structure soft | Yes — dust lanes in M20 Trifid are soft, M8 hourglass is mushy | **Sharpening doctrine** | `deconvolution` skipped (should have run: SNR 74.9 ≫ 50, FWHM 1.87px is marginally under-sampled but RL would still help). Log notes "Any further saturation might push into artificial-looking territory" — the agent self-stopped. | Overall image flat/low contrast | Present | **Tonal doctrine** | `curves_adjust` was available but never called. MTF or GHT curves could have reshaped mid-tones after star removal. `local_contrast_enhance` was called but undermined by the wrong-file bug. |

### The checkpoint bug (framework issue)

In the nonlinear phase, at step 3 the agent called `save_checkpoint("starless_base")` — but it did so _after_ the starless image already existed on disk as a separate file. In muphrid, `save_checkpoint` stores `state.paths.current_image`, which at that moment was still pointing at the pre-starless vibrant_nebula file (the star_removal tool wrote a new starless file but did not necessarily update `current_image` — or the agent misremembered which was current). The agent then ran T17 LCE, got an output that still had stars, noticed the problem, restored the checkpoint (which was the pre-starless state), and got confused in a loop it couldn't escape cleanly. The log shows three `restore_checkpoint("starless_base")` attempts and a `commit_variant` that rationalized the failure and moved on.

**This is a framework failure**, not a tool failure. `save_checkpoint` saves the wrong thing relative to the agent's mental model, and the tool names don't guard against it.

### Two upstream red flags the agent should have caught but didn't

- **Gradient removal made things worse and got accepted.** Baseline gradient_magnitude was 0.012 (essentially flat). GraXpert s=0.6 pushed it to 0.462, s=0.8 pulled back to 0.428. Both are **38x worse than baseline**. The phase prompt explicitly says "If signal_coverage dropped significantly from the baseline, the background model subtracted the target — increase smoothing and re-run." The agent noticed (log: "strong light pollution gradient from top-left") but diagnosed the situation _backwards_ — the pre-gradient image was fine; GraXpert over-fit on the nebula. Correct action: revert to pre-gradient image and skip the step. Available action: none — there is no "skip" tool and no polynomial fallback.
- **15 lights is light for pro.** ~150 min of integration (best case) isn't long enough for the faint outer Hα and blue reflection structure to climb safely above the noise floor. The agent never flagged dataset adequacy. There is no tool to assess "is this enough integration time for the target type?"

## 3. Can a Skilled Human Reach Pro Results With These Tools Alone?

I audited every tool file under `muphrid/tools/`. Here is the honest verdict.

### Strong enough for pro work

| Capability | Tool | Notes |
|---|---|---|
| Calibration (bias/dark/flat) | `build_masters`, `calibrate` | Full Siril feature exposure; winsorized/sigma/linear-fit rejection all accessible. |
| Registration | `siril_register` | Homography, two-pass, tunable findstar; Lanczos4 interpolation. |
| Frame selection | `select_frames` | Multi-criterion (FWHM/roundness/star count/background). |
| Integration | `siril_stack` | Full weighting modes (wfwhm/ws/wnoise), normalization options, rejection families. |
| 2D statistics & diagnostics | `analyze_image` | **Genuinely pro-grade.** photutils Background2D, MAD + wavelet MRS dual noise, IRAFStarFinder, gradient magnitude, per-channel, linearity estimator. Better than what most hobbyists run. |
| Color calibration | `color_calibrate` | PCC + SPCC, platesolve auto, per-filter SPCC, atmospheric extinction, background tolerance. |
| Deconvolution | `deconvolution` | RL + Wiener + three PSF sources (stars/blind/manual) + TV/Hessian regularization + NaN detection. |
| Wavelet multiscale | `multiscale_process` (T27) | **The single most impressive tool.** True B3-spline à trous with per-scale sharpen/denoise/suppress/passthrough, MAD soft-thresholding, ratio/additive color reconstruction, mask blending. Parity with PixInsight MLT. |
| Masking | `create_mask` (T25) | Luminance/inverted/range/channel_diff with morphology and feathering. Excellent. |
| Star separation | `star_removal` | StarNet2 + upscale heuristic + color-mode check. |
| Stretch | `stretch_image` | GHS (D/B/SP/LP/HP/clip_mode/color_model) + asinh + autostretch. Variant fingerprinting prevents dupes. |
| Pixel math | `pixel_math` | Siril `pm` with 10-var support and mono/RGB broadcast — enough for HDR blends. |

### Adequate but limited (workarounds exist)

| Gap | Current tool | Workaround | Friction |
|---|---|---|---|
| Arbitrary tone curves (multi-anchor on R/G/B/L/a/b/H/S) | `curves_adjust` — MTF (3 points) or GHT (parametric) | Chain MTF + GHT + per-channel MTF | Medium. Achievable but verbose. |
| HDR compositing (MaskedStretch / HDRMultiscale) | None directly | `stretch_image` × 2 variants + `create_mask` + `pixel_math` blend | **High.** 4 tool calls for one conceptual action. Agent rarely attempts it. |
| Masked-only processing of any tool | None | Run tool globally → `create_mask` → `pixel_math` blend original+processed via mask | **High.** Agent has to assemble manually every time. |
| Saturation on arbitrary hue ranges | `saturation_adjust` hue_targeted (7 Siril indices) or GHT_sat (global) | Extract channels via pixel_math, boost, recombine | Medium–High. |
| Polynomial gradient fallback (DBE/ABE equivalent) | `remove_gradient` — GraXpert AI only | None | High. When GraXpert over-fits on target signal (as on M20), there is literally no alternative. |
| Star reduction with morphological precision | `reduce_stars` exists (not read in this audit but present) | Use T26 | Low if agent remembers T26 exists. |
| HSV/HSL curves | None | Possible via numpy + custom pixel_math | High. |
| Integrated masked_apply | None | 3-step T25 + tool + T23 | High. Biggest ergonomic gap. |

### Genuine missing capabilities

These cannot be achieved even with workarounds:

1. **Polynomial background extraction** (Siril's native `background -gradient -poly N` or an ABE-style morphological opening). Critical when GraXpert over-fits. A 30-line addition.
2. **Custom HSV-space tone curves.** Separate H, S, V curves applied in HSL or HSV space. Needed for nuanced color grading of emission nebulae and to desaturate over-saturated cores without killing mid-range saturation. ~120 lines using `skimage.color.rgb2hsv` + curves.
3. **Range-masked processing wrapper.** A meta-tool `masked_apply(tool, mask, blend_alpha)` that runs any processor inside a T25 mask without the agent constructing the pixel_math blend by hand. Could be a one-page wrapper.
4. **Drizzle-style upscaling stacking** (for undersampled data like this M20 dataset at FWHM 1.87 px). Siril supports drizzle; muphrid does not expose it. ~40 lines in `siril_stack` to add `-drizzle=2`.
5. **Automated HDR composite tool.** A single tool that takes the linear master, runs two stretches (faint-optimized + core-protected), auto-generates a range mask on the core, and blends. 3-stretch or 4-stretch variants. ~200 lines.

None of these is a blocker for pro results on typical targets — they are quality-of-life amplifiers. Priorities 1, 2, 3 are highest leverage.

### Clearly underexposed pro capabilities already in the codebase

Two pro-grade tools in muphrid are under-used because nothing in the prompt demands them:

- **T27 multiscale_process** is capable of selective-sharpening nebula detail at scales 2–3 while simultaneously denoising scales 0–1, with a luminance mask to protect the background. In the M20 run it was never called. The agent reached for T17 local_contrast_enhance instead. Doctrine issue.
- **T25 create_mask** was not called once in the M20 run. No masks, no selective processing, no HDR blend. Doctrine issue.

## 4. Framework & Prompt Layer — Higher-Leverage Than Tool Gaps

### 4.1 The quality rubric is metric-minimum, not picture-maximum

The system prompt says: _"Your work succeeds when metrics improve or hold through each transformation."_ That's a floor, not a ceiling. It produces agents that stop when nothing is obviously broken, rather than agents that push until the picture sings. The prompt never says "compare your current image to what a well-processed version of this target type looks like" — the agent has no visual reference model.

### 4.2 The "no narration, tool calls only" pressure is counterproductive for this task

From prompts.py line 141: _"If you respond with text instead of a tool call, you will be redirected to act."_ And line 138: _"Do not summarize what you just did for the human's benefit."_ These instructions were likely added to fight models that preamble endlessly. On the other hand, astrophotography processing is _exactly_ the kind of task where a good operator pauses, thinks about what they're seeing, and plans a strategy. The prompt discourages reflection and rewards action. Kimi K2.5, DeepSeek V3, and even Sonnet 4.6 all think better when allowed to reason between actions. In the M20 run, the agent's "reasons advanced" entries in the log show that when it _did_ reason, the reasoning was often good — but the system pushed it to execute fast instead of think slow.

### 4.3 Phase-level target doctrine is shallow, especially for NONLINEAR

The `LINEAR` phase prompt is excellent. It goes deep on gradient, color, green, denoise, decon with explicit dependency chains, guidance on smoothing for gradient, decision rules for skipping. By contrast, the `NONLINEAR` prompt is short — less than half the length — and touches each sub-topic lightly. Examples of what's missing:

- No discussion of when to use `multiscale_process` vs `local_contrast_enhance`.
- `create_mask` gets one mention, as a luminance mask for nebula sharpening. No mention of range masks, inverted masks, channel_diff masks for emission-line isolation.
- `curves_adjust` is triggered only "contrast_ratio below 0.3" — trivially narrow. In practice curves are used for midtone shaping, star color rescue, color grading, not just low-contrast recovery.
- HDR compositing gets a single paragraph in the `STRETCH` prompt and zero reinforcement in `NONLINEAR`.
- Iteration doctrine (multi-pass saturation, multi-pass curves, multi-stage sharpening) is not prescribed.
- No target-type branching in the nonlinear phase. The target-type guidance in SYSTEM_BASE covers emission nebulae / galaxies / globulars at a high level but doesn't translate into _nonlinear_ playbook ("for emission nebulae: star_removal → multiscale_process with nebula-luminance mask for scales 2–3 → curves_adjust to lift midtones → saturation_adjust hue_target=0 → saturation_adjust hue_target=3 → curves_adjust gentle S-curve → star_restoration → reduce_stars if needed").

### 4.4 Checkpoint semantics are easy to misuse

`save_checkpoint(name)` snapshots `state.paths.current_image`. If `current_image` doesn't point at what the agent _thinks_ it does (because a recent tool wrote a new file but didn't update `current_image`, or because the agent's mental model drifted), `save_checkpoint` locks in the wrong reference. The M20 run shows this failing silently.

Easy mitigations:
- `save_checkpoint` should echo the exact file path it is bookmarking, and prompt the agent to verify.
- `save_checkpoint` should refuse to save when `current_image` is stale (e.g. last tool call produced a different output path that `current_image` wasn't updated to).
- `restore_checkpoint` should show the agent the path it restored to and the diff since the last checkpoint.

### 4.5 Feedback loop doesn't treat regressions as hard failures

The agent accepted gradient_magnitude going from 0.012 to 0.428 because the "compare to baseline" step was advisory. Proposal: after `remove_gradient`, if a key metric regresses past some threshold vs. baseline (say 2x worse), the tool should return a _hard advisory_ in its result payload — not a warning line the agent can ignore, but a structured field like `"regression_warning": {...}` that the phase prompt instructs the agent to act on.

### 4.6 The agent has no long-term reference library

`memory_search` exists but in the M20 run it wasn't enabled (settings.json shows `"memory": false`). Even when enabled, there is no curated library of _"this is what a good M20 looks like; these are parameters that worked"_ — only past session memory. A seed library of known-good parameter sets per target type would raise the floor dramatically for first-time processing.

## 5. Prioritized Recommendations

These are sequenced by expected impact on final image quality, not by implementation effort.

**P0 — Prompt doctrine overhaul (highest leverage)**
- Rewrite the quality rubric in SYSTEM_BASE to include visual-reference reasoning, not just metric preservation.
- Expand the NONLINEAR phase prompt 3–4×. Include an explicit, non-prescriptive playbook per target type. Include iteration doctrine. Make `multiscale_process` + `create_mask` the default sharpening path, not `local_contrast_enhance`.
- Soften the "no text without tools" instruction. Replace with "think briefly between tool calls when the decision is nontrivial; don't narrate for the user."
- Add an explicit target-visibility checklist at the nonlinear → export boundary ("can the viewer see X, Y, Z structures that should be there for this target type?").

**P1 — Masked-apply wrapper (`masked_process`) tool**
- Single-tool wrapper: takes a primary tool name, its params, and a mask (either an existing path or a `create_mask` spec). Runs the primary tool on the current image, then blends `processed * mask + original * (1 - mask)` automatically. Removes the 3-step friction that is currently killing masked workflows in practice.
- Estimated effort: ~150 lines. See §7 for a patch stub.

**P2 — Polynomial gradient fallback in `remove_gradient`**
- Add `method: Literal["graxpert", "polynomial", "none"]` to `RemoveGradientInput`.
- Polynomial backend uses `siril: background -gradient -poly N` (N=1–3).
- When GraXpert regresses (post-tool gradient_magnitude > 2× pre-tool), return a structured `regression_warning` so the agent knows to consider `method="polynomial"` or revert.
- Estimated effort: ~70 lines. See §7.

**P3 — Per-channel stretch + masked stretch**
- Extend `GHSOptions` with optional `per_channel_D: list[float] | None` and `per_channel_SP: list[float] | None`. After weak color calibration this is the cleanest lever to fix residual color casts.
- Add `stretch_image(... mask_path=None)` — when present, apply stretch only inside the mask, blend with original outside. One-tool MaskedStretch.
- Estimated effort: ~80 lines.

**P4 — Richer curves (`curves_adjust` method=`points`)**
- Accept `points: list[tuple[float, float]]` for piecewise-linear or cubic-spline custom curves per channel.
- Implement in numpy; bypass Siril MTF's 3-point limit.
- Estimated effort: ~120 lines.

**P5 — HSV-space saturation curves**
- New tool `hsv_adjust(h_curve, s_curve, v_curve, mask_path)` using `skimage.color.rgb2hsv/hsv2rgb`.
- Fixes the "only 7 hue indices" limitation of current saturation_adjust.
- Estimated effort: ~130 lines.

**P6 — Checkpoint hardening**
- `save_checkpoint` returns the resolved file path it saved and its stat summary (size, hash). The returned message tells the agent exactly what was bookmarked.
- Optional `save_checkpoint(explicit_path=...)` to decouple from `current_image`.
- `restore_checkpoint` returns a diff summary.

**P7 — Drizzle option on `siril_stack`**
- Expose `-drizzle=N` (N=2 or 3) for undersampled data like the M20 set (FWHM 1.87 px). Adds resolvable detail for fast short-focal-length setups.
- Estimated effort: ~30 lines.

**P8 — Dataset-adequacy advisory**
- Add to `analyze_frames` (or a new `assess_dataset` tool) a structured `integration_assessment` block: total integration time, target-type baseline expectation, estimated "quality ceiling" for the current integration. Warns the agent + user when the dataset is simply insufficient for the ambition of the session. Doesn't refuse to process — informs expectation.

**P9 — Seed memory with target-type parameter libraries**
- Even without real user history, pre-populate `memory` with curated parameter sets for common targets (M42, M31, M20, M8, NGC 7000, Horsehead, etc.) including GHS params, saturation amounts, multiscale weightings that an expert would start from. Gives `memory_search` a real knowledge base on day one.

**P10 — Visual reference comparison in HITL**
- At the nonlinear HITL gates, optionally show a reference image (a known-good processed version of the target, if one exists in memory) next to the current state. Lets the user and the agent reason against a visual target, not just against numbers.

## 6. "Scripted-ness" Audit

The user specifically asked that nothing be formulaic or scripted. Here's where rigidity hurts:

| Area | Scripted? | Assessment |
|---|---|---|
| Preprocessing (T01–T08) order | Yes | Correct — these have hard physical dependencies. |
| Linear tools (T09–T13) order | Partially flexible — dependency chain is advisory | Correct. |
| Each linear tool's internal behavior (one backend per tool) | Yes | **Too rigid.** `remove_gradient` should have polynomial fallback; `noise_reduction` should have wavelet alternative to GraXpert. |
| Stretch parameter space | Flexible | Good. |
| Nonlinear tool selection | Flexible | Good in principle, but doctrine voids don't give the agent guidance to pick the right tool for the job. |
| Masked processing | Scripted (3-step chain) | **Too rigid.** Solved by P1 wrapper. |
| Checkpoint model | Opaque & scripted | P6 fix. |
| HDR compositing | Scripted (4-step chain) | Solved by masked wrapper + richer prompt. |
| Variant selection | Manual | Automatable; low priority. |

## 8. Model-Size Reading of the Evidence

Kimi K2.5 did _not_ fail this run on raw capability — it reasoned clearly in the log, noticed real problems, adjusted parameters, asked intelligent questions at HITL gates. Where it fell short: it didn't reach for masked workflows, didn't use T27, didn't call deconvolution, stopped saturating too early, accepted the gradient regression, got stuck in the checkpoint confusion. Those are all doctrine failures, not reasoning failures.

A stronger model (Opus 4.6, Sonnet 4.6) on the _same_ system would likely produce a marginally better result because it would push harder on saturation, use more passes of curves, maybe try HDR compositing — but it would still be throttled by the same prompt that rewards metric-minimum and the same 3-step friction on masked operations. The gain from model size alone is probably ~15% of the gap to pro. The gain from doctrine + P1 + P2 is probably 40–60%. The tool gaps (P3–P5) add the last 10–20%.

**Upgrade the prompts and the masked-apply wrapper first. Then re-evaluate with the same model. Then consider a model upgrade.**

## 9. Verification & Scope Caveats

- LangSmith was not available in this session (no API key accessible from my sandbox). All findings are grounded in the on-disk run artifacts, full code read, and the processing_log.md.
- I did not run any of the tools myself, so performance claims about GraXpert polynomial fallback, drizzle, etc. are design-level, not measured.
- Patch stubs are design sketches, not drop-in PRs — they reflect API shape and intent. The checkpoint bug in particular deserves a reproduction test before patching.
- The ~85% / 15% tool-coverage number is judgment, not measurement. It's a useful mental model, not a contract.
